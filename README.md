# Map File Fundamentals

A map file is a comprehensive text output generated by the linker during the final build stage. It serves as a "blueprint" or bridge between the **Logical Sections** defined in C code and the **Physical Memory** defined in the linker script.

## Primary Purposes

- **Debugging and Optimization**: Identifying the exact location of functions and variables.
- **Size Analysis**: Analyzing firmware placement and footprint.
- **Symbol Mapping**: Detailing exactly where every function and variable lives in memory.

The GNU (GCC) map file is generally divided into three phases:
- **Archive Member References**: Shows object files pulled from libraries to satisfy symbols.
- **Memory Configuration**: Reflects the physical layout (SRAM vs. Flash) of the chip.
- **Linker Script and Memory Map**: The core of the file, detailing every function/variable's location, size, and source file.

## Physical Memory Configuration (S32K144)

The S32K144 typically features **512 KB of Flash** and **64 KB of SRAM**. Your configuration defines the following boundaries:

| Memory Region   | Origin      | Length | Description                          |
|-----------------|-------------|--------|--------------------------------------|
| `m_interrupts`  | `0x00000000`| 1 KB   | Interrupt Vector Table (Flash)       |
| `m_flash_config`| `0x00000400`| 16 B   | MCU security and configuration bits  |
| `m_text`        | `0x00000410`| ~511 KB| Main Application Program Code (Flash)|
| `m_data`        | `0x1FFF8000`| 32 KB  | SRAM (Lower bank/SRAM_L)             |
| `m_data_2`      | `0x20000000`| 28 KB  | SRAM (Upper bank/SRAM_U)             |

## Standard GNU Sections and Placement

The linker organizes your code and data into specific sections based on their mutability and initialization.

### A. Read-Only Sections (FLASH/ROM)
These sections reside in the program space and cannot be modified at runtime.

- **`.interrupts`**: Resides at the very beginning (`0x00000000`). Contains the `__VECTOR_TABLE` used by the CPU to locate handlers like `Reset_Handler`.
- **`.FlashConfig`**: A critical 16-byte section at `0x00000400`. Misconfiguration here can permanently lock the MCU.
- **`.text`**: Contains executable machine code (compiled functions), startup code, and interrupt vectors. In CAN_Stack project, this occupies approximately **20 KB**.
- **`.rodata`**: "Read-Only Data" for constants like hardcoded strings or constant configuration structures.

### B. Read-Write Sections (RAM)
These sections handle mutable data and temporary runtime storage.

- **`.data`**: Contains variables with non-zero initial values (e.g., `int16_t foo = 1337;`). Stored in Flash (LMA) but copied to SRAM (VMA) during startup for modification.
- **`.bss`**: "Block Started by Symbol." Reserved for variables initialized to zero or uninitialized (e.g., `int16_t bar;`). They occupy no space in Flash but are cleared in SRAM at startup.
- **`.stack`**: Reserved area for local variables and function call return addresses. In your file, it sits at the end of SRAM (`0x20007000`) and grows downwards.
- **`.heap`**: Reserved for dynamic memory allocation (`malloc`). Starts after the `.bss` section.

## Notable CAN Stack Observations

- **Binary Footprint**: Application is lean, using roughly **22 KB of Flash**, leaving most of the 512 KB available.
- **RAM Vector Table**: The presence of `.interrupts_ram` at `0x1FFF8000` suggests the vector table is copied to RAM at runtime for faster interrupt latency or dynamic changes.
- **CAN Impact**: Specific CAN-related variables, such as the Tx1KB buffer and `CanTp_TxRt` state variables, are located in the `.bss` section.

## Section Categories

### 1. Non-Volatile Memory (Flash/ROM)
These sections stay in the chip even when the power is off. They are "Read-Only" during normal execution.

| Section          | Description |
|------------------|-------------|
| `.text`          | The actual executable machine code of your program. |
| `.rodata`        | Constant variables (e.g., `const int x = 10;`) and string literals. |
| `.isr_vector`    | The Interrupt Vector Table (IVT). Contains addresses for the Reset handler and peripheral interrupts. Must be at a specific address (usually `0x00000000`). |
| `.init` / `.fini`| Executable code for process initialization and termination. |
| `.init_array` / `.fini_array` | Tables of pointers to functions that need to be called at startup (like C++ constructors) or exit. |
| `.FlashConfig`   | Specific to certain MCUs (like NXP); stores security bits or device configuration bytes. |
| `.ARM`           | Contains index tables for exception handling and stack unwinding specific to the ARM architecture. |

How to Run the GCC Map Parser
This parser analyzes GCC generated .map files to help you optimize embedded memory usage.

Make sure you have Python 3.x installed.

To run the parser, open your terminal or command prompt and use the following command structure:
python <map_parser.py> <sample.map>
